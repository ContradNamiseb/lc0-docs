39 // Various shaders used by the backend:
40 //
41 // 1. Expand planes: Used to convert packed bit-board representation to
42 // 'planes' that is input to NN
43 // 2. Winograd Input transform.
44 // 3. Winograd Output transform.
45 // - Fused with bias add, skip connection add, and relu.
46 // - Fused with SE, bias add, skip connection add and relu.
47 // 4. Policy Map layer. (can also be done on CPU side)
48 // 5. 1x1 convolution custom kernel (used by policy and value heads).
49 // - TODO: Try replacing this with conv metacommand when available.
50 //
51 // For best performance it would seem that we would need two copies of all the
52 // shaders - fp16 and fp32 versions. However 2 copies not needed for now, as:
53 // i) We should use typed UAVs for resource access as they seem to be
54 // faster. With Typed UAVs, the shader automatically recieves datatype converted
55 // values (e.g: in fp32 even when the allocation was in fp16)
56 // ii) Most of these operations are memory bound - except for the GEMM, but:
57 // iii) Due to driver/compiler bugs or lack of optimizations fp16 path seems
58 // slower on both Nvidia and AMD for most of the shaders - even GEMM on AMD
59 // is way slower with fp16 math than with fp32.
60
61 // Only expand planes has different shaders for different datatypes.
62 // - Mostly a meaningless 'early' optimization as this shouldn't be the bottleneck.
63 ID3D12PipelineState* expand_planes_fp16_;
64 ID3D12PipelineState* expand_planes_fp32_;
65
